下载不是胜利。**用户第一次打开后必须快速看到价值**；付费墙应该出现在价值被感受到之后，而不是挡在门口。

这一小时把转化漏斗里**最容易翻车的 4 段**优化一遍。

## 🎯 你将完成

- 一份**前 90 秒漏斗诊断**（哪里在流失）
- 优化后的**付费墙触发时机**
- 一个明确的**激活事件**定义
- 一张**留存看板**：每周看哪 5 个数

## 第 1 步：前 90 秒漏斗诊断（0–10 分钟）

### 在 PostHog 拉这张图

用 Funnels 功能，按顺序加事件：

```
1. app_opened
2. onboarding_completed
3. core_action_done
4. paywall_shown
5. trial_started
```

时间窗设 **24 小时**（不是默认的 30 天，新用户当天没行动后面也不会）。

### 健康基线

| 段 | 通过率 | 含义 |
|---|---|---|
| `app_opened → onboarding_completed` | ≥ 70% | Onboarding 不太长 |
| `onboarding_completed → core_action_done` | ≥ 50% | 用户找到核心入口 |
| `core_action_done → paywall_shown` | ≥ 60% | 付费墙触发时机 OK |
| `paywall_shown → trial_started` | ≥ 30% | 价值主张清楚 |

### 找瓶颈

哪段最低就先修哪段。**不要 4 段同时改**，否则你看不出哪个变量起的作用。

## 第 2 步：优化前 90 秒（10–25 分钟）

### Onboarding 流失太高（< 50%）

常见原因：

- **太长**：超过 5 步用户就跑
- **要权限太早**：第 1 步就弹通知 / 位置权限，触发警觉
- **没说清这是什么**：第 1 屏不复述广告承诺，用户以为下错了

修复：

1. 砍到 **4 步**：欢迎 → 痛点共鸣 → 价值演示 → 权限（如需要）
2. 权限放最后一步，配合"我们只用它来 X"的解释
3. 第 1 屏的标题**字号 ≥ 40px**，复述广告里那句话

### 找不到核心动作（< 30%）

常见原因：

- 主屏幕有 5 个 tab，用户不知道点哪
- 核心动作藏在二级菜单
- 默认状态是空白，用户不知道该输入什么

修复：

1. 主屏幕**只留核心动作按钮**，其他放设置里
2. 给用户一个**预填示例**或 placeholder（"试试输入：午睡 20 分钟"）
3. 第 3 次打开还没用过核心动作 → 弹一个**单步教程**

## 第 3 步：付费墙时机（25–40 分钟）

### 核心心法

**别把付费墙挡在门口。** 先让用户**体验到核心价值**，再触发付费墙。

### 三种触发模式

| 模式 | 适合 | 转化率 |
|---|---|---|
| **Hard wall**（不付费用不了任何核心功能） | 一次性工具、AI 类 | 高但留存差 |
| **Soft wall**（免费有限次数，付费解锁全部） | 习惯类、内容类 | 平衡 |
| **Feature gate**（核心免费，高级付费） | 工具类、生产力类 | 转化低但 NPS 高 |

### 推荐：首次使用免费 + 第 N 次触发

```
第 1 次完成核心动作 → 不弹付费墙，让用户感受成功
第 1 次结果页 → 弹"想要 X 高级功能吗？"软推
第 2 次完成核心动作 → 弹付费墙
```

### 试用期：先测 3 天，不要 7 天

新手以为**试用期越长越好**——错的。

- 7 天试用 → 用户忘了取消，但你的 unsubscribe rate 很高
- 3 天试用 → 紧迫感强，付款率高，长期 LTV 反而更高
- 同时跑两组测，看 30 天后净付费用户

### 付费墙文案：用具体损失

| ❌ 模糊 | ✅ 具体 |
|---|---|
| 解锁全部功能 | 解锁连续打卡。错过一天就重新开始攒 |
| 升级到 Pro | 升级 Pro：今天就能用 Widget 一眼看本周进度 |
| 高级订阅 | 一年订阅：每天 8 分钱，省一杯咖啡的钱 |

## 第 4 步：定义激活事件 + 留存看板（40–50 分钟）

### 什么是"激活事件"

激活事件 = 用户**真正感受到价值**的那一刻。不是注册、不是 onboarding 完成。

| App 类型 | 激活事件示例 |
|---|---|
| 习惯追踪 | 完成第 3 次记录 |
| AI 工具 | 第 1 次生成成功的结果 |
| 内容工具 | 第 1 次保存项目 |
| 计时器类 | 第 1 次完整跑完一轮 |

### 每周看的 5 个数

在 PostHog 建 Dashboard，每周一上午 9 点看：

```
1. 新增激活用户数（不是下载数）
2. D1 留存：今天激活的用户，明天还在用的比例
3. D7 留存：本周一激活，下周一还在用
4. 试用转付费率
5. 月退订 / 月新订比例
```

### 健康基线

- **D1 留存** ≥ 40%（< 30% 是产品问题）
- **D7 留存** ≥ 20%（< 10% 是定位问题）
- **试用 → 付费** ≥ 5%（< 3% 是付费墙问题）

---

## 第 5 步：本周 1 个实验（50–60 分钟）

### 一周只改一个变量

新手最常错：周一改了 onboarding，周三改了截图，周五改了付费墙——**数据动了你也不知道是哪个起的作用**。

### 选实验

从你诊断出的**最瓶颈**那一段，挑一个变量：

| 瓶颈段 | 实验变量 |
|---|---|
| Onboarding 流失 | 减一步 / 改第 1 屏文案 / 推迟权限 |
| 核心动作流失 | 加预填示例 / 改主屏布局 |
| 付费墙转化 | 改触发时机 / 改试用期 / 改文案 |
| 试用 → 付费 | 改价格 / 改对比表 / 改紧迫感 |

### 跑多久看结果

- 流量 ≥ 100/天 → 跑 **5 天**
- 流量 30-100/天 → 跑 **10 天**
- 流量 < 30/天 → 这个阶段流量太小，先做 [冷启动](/zh/tutorials/app-cold-start/)

### 实验日志

记一行：

```
2026-W27: 试用从 7 天改 3 天
  Before: 30% trial→paid, 退订率 12%
  After:  ____, ____ （留空，跑完再填）
  决策: ____
```

- ✅ 一张前 90 秒漏斗截图，知道哪段在流失

- ✅ 一个明确的激活事件定义
- ✅ 一张 PostHog 留存看板
